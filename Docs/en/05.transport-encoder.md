# Transport & Encoder

> 中文版: [05.transport-encoder.md](../zh/05.transport-encoder.md)

Both the **transport** and the **encoder** in GoPlay are orthogonal, swappable interfaces. Server and Client pick a transport through a generic parameter; the encoder is selected by `EncodingType` (or injected directly). You can swap TCP &rarr; WebSocket or Protobuf &rarr; Json without touching business code.

## Overview

```text
  Business Processor
        │
   ┌────┴────┐
   │ Server  │         ← partial class, handles only Route/Processor semantics
   └────┬────┘
        │   generic T : TransportServerBase
   ┌────┴────┐
   │Transport│         ← TcpServer / NcServer / WsServer / WssServer
   └────┬────┘
        │
     socket / websocket
```

```text
  Package (framework frame)
        │
   ┌────┴────┐
   │ Encoder │         ← IEncoder: Protobuf / Json; header and body share the same encoding
   └────┬────┘
        │
   wire bytes
```

## Transport Abstractions

### Base Classes

- [`TransportServerBase`](../../Frameworks/Core/Transports/Base/TransportServerBase.cs): server-side abstraction, exposes `OnClientConnected/Disconnected/Error/DataReceived` events and `Start/Stop/Send/DisconnectClient`.
- [`TransportClientBase`](../../Frameworks/Core/Transports/Base/TransportClientBase.cs): client-side abstraction, provides `ConnectAsync/Disconnect/Send/Recv/SupportPush`.

### Push vs Pull Semantics

The key flag `SupportPush` (default `true`) picks the receive path:

- **Push**: the transport calls back `OnDataReceived` / `OnDataReceivedSpan` **proactively** when data arrives. NC / Ws / Wss fall here; based on IOCP or event callbacks, **no Recv loop needed**.
- **Pull**: the framework spins a `RecvLoop` that keeps calling `Transport.Recv`. Built-in `TcpClient` / `TcpServer` fall here; fits raw-socket implementations.

`Client.cs` routes based on `Transport.SupportPush`: Push runs only `SendLoopAsync` + `TimeoutLoop`; Pull adds a `RecvLoop`.

### Four Built-in Implementations

| Implementation | Assembly | Underlying | Semantics | When to Use |
|----------------|----------|------------|-----------|-------------|
| `TcpServer` / `TcpClient` | `GoPlay.Core` (built-in) | `System.Net.Sockets.TcpClient` | Pull | teaching / small services |
| `NcServer` / `NcClient` | `GoPlay.Core.Transport.NetCoreServer` | [NetCoreServer](https://github.com/chronoxor/NetCoreServer) | Push | **recommended for TCP production** |
| `WsServer` / `WsClient` | `GoPlay.Core.Transport.Ws` | NetCoreServer WebSocket | Push | browser / mini-game |
| `WssServer` / `WssClient` | `GoPlay.Core.Transport.Wss` | NetCoreServer Secure WebSocket | Push | production browser (TLS) |

Pick one:

```csharp
var server = new Server<NcServer>();              // TCP
var server = new Server<WsServer>();              // WebSocket
var server = new Server<WssServer>();             // WSS

var client = new Client<NcClient>();              // matching client
```

## Wire Frame

```text
  +-----------+-----------+---------+---------+
  | outerLen  | headerLen | Header  |  Body   |
  |  ushort   |  ushort   | bytes   |  bytes  |
  +-----------+-----------+---------+---------+
              \____________ innerLen ________/
```

- All lengths are little-endian `ushort` (<=65535).
- Packages beyond that must chunk - see [10.protocol.md](./10.protocol.md#chunk--fragmentation).
- Entry points: `WriteTo(IBufferWriter<byte>)` and `ParseRaw(ReadOnlySpan<byte>)` in [Frameworks/Core/Protocols/Package.cs](../../Frameworks/Core/Protocols/Package.cs).

### Zero-Allocation Hot Path

Send side does a full zero-copy frame build:

1. `SendChannel` delivers a `Package`; `SendLoopAsync` calls `pack.WriteTo(writer)`, which writes `[outerLen][headerLen][header][body]` in place into an `ArrayBufferWriter<byte>` **reused for the whole connection**.
2. `IEncoder.EncodeTo` plumbs Protobuf's `IMessage.WriteTo(IBufferWriter<byte>)` so header and body land in the same span without intermediate `byte[]`.
3. `Transport.SendAsync(writer.WrittenMemory, ct)` sends once; transport subclasses must copy the bytes into their own pending buffer **before returning the ValueTask**, so the caller can reset the writer.

Receive side is symmetric: [`TransportServerBase.InvokeOnDataReceivedSpan`](../../Frameworks/Core/Transports/Base/TransportServerBase.cs) hands a frame as `ReadOnlySpan<byte>` to the server, and `Package.ParseRaw(ReadOnlySpan<byte>)` decodes it synchronously.

## Encoder Abstraction

### Interface

```csharp
// Frameworks/Core/Interfaces/IEncoder.cs
public interface IEncoder
{
    EncodingType Type { get; }
    void AssertTypeValid<T>();

    byte[] Encode<T>(T data);
    T Decode<T>(byte[] data);

    int GetEncodedSize<T>(T data);
    int EncodeTo<T>(T data, Memory<byte> dest);
}
```

- `GetEncodedSize`: predicts length on the hot path; `EncodeTo` then writes in place into an `IBufferWriter` slice. Protobuf uses `IMessage.CalculateSize()` for zero allocation; Json falls back to `Encode(data).Length`.
- `EncodeTo`: paired with `GetEncodedSize`; Protobuf implementation plumbs `MessageExtensions.WriteTo(IBufferWriter<byte>)` straight into the span.

### Two Built-in Implementations

| Encoder | EncodingType | Pros | Use case |
|---------|--------------|------|----------|
| `ProtobufEncoder` | `Protobuf = 0` (default) | fast, compact, zero-alloc hot path | production |
| `JsonEncoder` | `Json = 1` (LitJson) | human-readable, scripting interop | debugging / embedded scripting |

Switch:

```csharp
var server = new Server<NcServer>
{
    EncodingType = EncodingType.Json,
};
```

Client and server must agree; the handshake freezes encoding for the whole session.

## Chunk / Fragmentation

- Your code only calls `Send(pack)`; `Package.Split()` inside `SendLoopAsync` splits oversized packages into multiple frames.
- All chunks share the same `Id`/`Route`; `ChunkCount` and `ChunkIndex` identify the fragments.
- The receiver reassembles in [Client.Chunk.cs](../../Frameworks/Client/Client.Chunk.cs)'s `ResolveChunk` by `Id`.
- Server and client logic is symmetric.

Protocol details in [10.protocol.md](./10.protocol.md#chunk--fragmentation).

## Custom Transport

Need KCP / QUIC / SSE / IPC? Implement two base classes:

```csharp
// Server
public class MyTransportServer : TransportServerBase
{
    public override bool SupportPush => true;     // prefer push semantics for new transports

    public override void Start(string host, int port, CancellationTokenSource cancelSource = null)
    {
        // start listener; on new connection call InvokeOnClientConnected(clientId)
        // for each complete framed chunk call InvokeOnDataReceivedSpan(clientId, span)
    }

    public override void Stop() { ... }
    public override void Send(uint clientId, byte[] data) { ... }
    public override ValueTask SendAsync(uint clientId, ReadOnlyMemory<byte> framedBytes, CancellationToken ct) { ... }
    public override void DisconnectClient(uint clientId, Exception err) { ... }
    public override string GetClientIp(uint clientId) { ... }
}

// Client
public class MyTransportClient : TransportClientBase
{
    public override bool SupportPush => true;
    public override Task<bool> ConnectAsync(string host, int port, TimeSpan timeout) { ... }
    public override void Disconnect() { ... }
    public override ValueTask Send(ReadOnlyMemory<byte> framedBytes, CancellationTokenSource cts) { ... }
    // on receive, invoke the handler set via SetDataReceivedSpanHandler
}
```

Usage is identical to built-in transports:

```csharp
var server = new Server<MyTransportServer>();
var client = new Client<MyTransportClient>();
```

Reference implementations: [`NcServer`](../../Frameworks/Transport.NetCoreServer/NcServer.cs), [`WsServer`](../../Frameworks/Transport.Ws/WsServer.cs) - production-grade push-semantics templates.

## Custom Encoder

Implement `IEncoder` and register into `EncoderFactory`. For example, MessagePack:

```csharp
public class MsgPackEncoder : IEncoder
{
    public EncodingType Type => (EncodingType)42;   // extend the enum or reuse one
    public void AssertTypeValid<T>() { ... }
    public byte[] Encode<T>(T data) => MessagePackSerializer.Serialize(data);
    public T Decode<T>(byte[] data) => MessagePackSerializer.Deserialize<T>(data);
    public int GetEncodedSize<T>(T data) { ... }
    public int EncodeTo<T>(T data, Memory<byte> dest) { ... }
}
```

Note: the Header itself is a Protobuf message. Swapping encoding for headers requires coordinated changes on both sides; swapping only the body encoding is a lighter path (and usually enough).
